## Your mission

Implement the Social Content Pipeline and basic Admin User Profiles in one cohesive build.
The plan file above is the source of truth — follow its schemas, API matrix, and phased delivery.

Goal: When an admin publishes a blog post, the system auto-generates platform-specific social
drafts (LinkedIn, X, Instagram, Facebook). Admins configure accounts and review batches;
clients edit drafts, copy fields to clipboard, and mark as posted externally.

Copy-paste workflow only — NO OAuth, NO platform API publish in MVP.

## Build order (strict — each step should compile)

### Step 1 — Identity & auth foundation
- Add src/auth/types.ts (Workspace, WorkspaceUser, UserPreferences, AuthContext)
- Add src/storage/workspace-users.ts (dual FS/Mongo; mirror humanizer-config pattern)
- Session auth: email + password for admins; keep site password for clients
- Bootstrap: first MASTER_KEY login creates workspace + owner user if none exists
- Extend middleware: requireAdmin, requireSiteMember (admin OR client with site access)
- Update editor Login.tsx: admin email login + client site password (keep both paths)
- Workspace.planTier field + UI badge (free | pro | agency) — no billing integration

Preference tiers (do NOT mix):
  Workspace → plan tier, BYOK API keys (existing IntegrationsStore), workspace defaults
  Site      → social accounts, humanizer tone/voice, style guide, Resend email
  User      → display name, lastSiteId, defaultHumanizerMode (embedded on WorkspaceUser)

Do NOT put API keys or social accounts on User profile.

### Step 2 — Social storage & types
- src/content/social-types.ts
- src/storage/social-config.ts   (SocialSiteConfig per site)

### Step 3 — Social generator (backend)

### Step 4 — Social API routes

### Step 5 — Social UI components

### Step 6 — Images (Phase 2 in plan — implement if time permits)

### Step 7 — Humanizer extension

## UX rules (non-negotiable)

Admin sees: setup, generate, batch review, full edit, humanize, copy, mark posted.
Client sees:  drafts list, edit caption + tags, copy panel, mark posted — nothing else.

Tags: store tags[] separately; never parse hashtags back out of bodyText.
Copy: per-field buttons, not one blob — users paste into platform editors piece by piece.
Re-run: same blog post can generate again; each run targets a different section/angle.
Published trigger: only when blog status is 'published', not draft saves.

---

## Explicitly OUT OF SCOPE (do not build)

- OAuth / platform API connections or auto-posting
- Per-user API keys (workspace vault only)
- Social accounts on user profile (per-site only)
- Billing / Stripe integration (planTier field only)
- Multi-workspace per user
- DALL-E / Flux image gen (Phase 3)
- Side-by-side diff view for social variants

## Acceptance criteria

- npm test && npm run build pass
- Admin can add per-site social accounts with include checkbox
- Publishing a blog post auto-creates a generation batch (when enabled + accounts included)
- Admin can review batch, accept 1–2 variants per platform into drafts
- Draft editor: caption, tag pills, per-field copy, humanize, mark as posted
- Client login sees Social → Drafts only; can edit, copy, mark posted
- Admin email login works; MASTER_KEY still works as bootstrap/break-glass
- Re-run generation from same blog post uses a different section angle
- API keys remain in workspace IntegrationsStore, not on user profile
