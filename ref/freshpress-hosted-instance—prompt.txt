══════════════════════════════════════════════════════════════════
FINAL ARCHITECTURE (target — do not contradict in later phases)
══════════════════════════════════════════════════════════════════

Three roles:
  Vendor (you)     → ship/host FreshPress, private source repo
  Agency buyer     → Owner login (MASTER_KEY), builds sites, invites end clients
  End client       → Client login (site ID + site password), edits one site only

Platform split:
  CMS (API + /editor)     → Railway or Fly.io (one app per agency buyer when hosted)
  CMS data                → MongoDB Atlas (one database per agency, e.g. freshpress_acme)
  CMS files (interim)     → Railway/Fly volume for DATA_DIR (publishes + wp media) until Blob migration
  Live client websites    → Buyer’s Vercel (static HTML via publish + VERCEL_TOKEN)
  Contact forms           → POST to CMS API (APP_URL) → buyer’s Resend (BYOK per site)

Why NOT Vercel for CMS:
  Current code uses long-running Express, local/volume disk for publishes and media, and
  long WordPress imports. Railway/Fly match the codebase with minimal refactor.

Auth (today — no user DB yet):
  Agency owner  → MASTER_KEY in env on CMS instance
  End client    → clientPasswordHash per site in Mongo; invite link APP_URL/editor/?site={id}

Critical product rule:
  CMS must be always-on at a public APP_URL so end clients can edit. A laptop-only CMS does NOT work
  for client editing unless tunneled 24/7.

PHASE 2 — Hosted CMS per buyer on Railway or Fly (Sprint 1)
══════════════════════════════════════════════════════════════════

Goal:
Make FreshPress deployable as ONE Railway or Fly app per agency buyer (vendor-provisioned).
Buyer and their end clients use https://{agency}.yourdomain.com/editor — no local install.

Preferred host order: Railway first (simpler DX), Fly.io as documented alternative.

Model summary:
- One Railway/Fly service per buyer (vendor creates; buyer does NOT get host dashboard by default)
- One MongoDB database per buyer on shared Atlas (MONGODB_URI → db name freshpress_{slug})
- Persistent volume mounted at DATA_DIR for publishes + WordPress media (until object storage sprint)
- MONGODB_URI required when HOSTED=1
- MASTER_KEY unique per instance; handoff URL + key to agency owner
- APP_URL = public CMS URL (editor invites + contact form API)
- Buyer’s VERCEL_TOKEN on instance (or Admin UI Sprint 2) deploys end-client sites to THEIR Vercel

Sprint 1 tasks:

2.1 Railway deployment packaging (primary)
- Add railway.toml or documented Railway setup (build: npm run build; start: npm start)
- Nixpacks/Node 20; root start runs dist/server.js
- Mount volume at /data → DATA_DIR=/data
- Public networking; health check GET /api/health
- Document equivalent fly.toml (Fly.io) as alternative in docs/HOSTED-INSTANCE.md

2.2 Hosted storage profile
- When HOSTED=1: require MONGODB_URI; warn/fail if filesystem-only for production
- Keep filesystem + volume for publish bundles and media (minimal change vs Vercel Blob sprint)
- Local dev unchanged (HOSTED unset, optional Mongo, ./data)

2.3 Env contract — docs/HOSTED-INSTANCE.md
Required per buyer CMS instance:
  HOSTED=1
  MASTER_KEY
  MONGODB_URI          # mongodb+srv://.../freshpress_{slug}
  APP_URL              # https://cms buyer public URL
  DATA_DIR=/data       # Railway/Fly volume
Optional BYOK (env in Sprint 1; Admin UI in Sprint 2):
  VERCEL_TOKEN         # deploy published client sites to buyer Vercel
  VERCEL_TEAM_ID
  ANTHROPIC_API_KEY / OPENROUTER_API_KEY

2.4 Manual provisioning — docs/PROVISION-BUYER.md
Checklist:
  1. Create Atlas DB freshpress_{slug}
  2. Create Railway (or Fly) service from repo template
  3. Attach volume; set env vars; generate MASTER_KEY
  4. Custom domain for APP_URL
  5. Deploy; verify /api/health and /editor
  6. Handoff: CMS URL + MASTER_KEY + BUYER-SETUP.md
  7. Log in instance registry (spreadsheet OK)

2.5 Legal + buyer docs
- COMMERCIAL-LICENSE.md
- docs/BUYER-SETUP.md — owner login, first site, client password, invite, Resend, publish to Vercel
- README deployment section already describes roles; link to BUYER-SETUP

2.6 Sprint 1 deferrals
- Object storage migration (Blob/R2) — optional later; volume is fine for Sprint 1
- Admin BYOK UI → Sprint 2
- Automated provision script → Sprint 2
- Multi-user agency team auth → later
- Stripe → Sprint 2

Sprint 1 exit criteria:
- [ ] Manual provision Buyer #1 on Railway in ~2 hours
- [ ] Agency owner: create site, publish to buyer Vercel, Resend form works
- [ ] End client: open /editor/?site=id, client login, edit + publish
- [ ] Live static site on Vercel survives CMS restart; forms work while CMS is up

Acceptance criteria:
- npm test and npm run build pass
- railway.toml (and fly.toml or doc) present
- HOSTED profile documented; local dev not broken
